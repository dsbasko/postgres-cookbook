# Story uplift 0/5 — Фаза 0 (снятие вето) и пилот

## Overview

Первый из шести ralphex-планов сюжетной доработки курса (сквозная история:
команда Brew с персонажами и диалогами поверх неизменной структуры знаний).
Канон истории — `docs/story-canon.md`, по-юнитная спека — `docs/story-scene-map.md`,
история решений — `docs/plans/completed/20260703-story-uplift.md`.

Этот план делает две вещи:

1. **Фаза 0 — снятие вето.** Действующее правило «функции, не персонажи»
   (скилл lecture-writer, канон подачи) заставит любого агента-переписчика
   выпиливать сцены. План штампует ратификацию канона истории, правит скилл,
   расширяет линтер и синхронизирует канон подачи.
2. **Пилот — жанровый срез.** Девять юнитов (модуль 00 целиком, 02-04, 02-06,
   04-01): все жанры лестницы §4.3 проверяются на реальных юнитах до массовой
   раскатки.

**ВАЖНО — запуск = ратификация.** Запуск этого плана владельцем есть акт
ратификации пяти пунктов §10 канона истории (отмена «функции, не персонажи»;
камео-управляющий Руслан, staff-Анна остаётся данными; жанр реплик протагониста; чат-жанр Павла строчными;
открытые вопросы приняты к сведению). Task 1 фиксирует это в каноне. Не
запускайте план, если не готовы подписаться под §10.

Следующий план семейства (`1-wave-i`) запускается только после приёмки тона
пилота владельцем (см. Post-Completion).

## Development Approach

- Правки прозы — только `i18n/{ru,en}/README.md`. Код юнитов, `query.sql`,
  `sqlc.yaml`, `Makefile` и секции `## Запуск` / `## Running it` (включая
  вставленный stdout) неприкасаемы.

- Роль тестов играют гейты: `check_unit.mjs --against-head`,
  `make web-check-coverage`, `make web-build`. Указанные в задаче гейты
  обязаны быть зелёными до перехода к следующей задаче.
- CRITICAL: при отклонении объёма обновлять этот план (➕ новые пункты,

- Коммит на задачу: тип `docs:` (проза, каноны) или `chore:` (скилл, линтер);

## Technical Details

**Порядок работ юнит-задачи:** прочитать канон целиком → запись юнита в карте
сцен → текущие RU/EN README → вписать RU-сцену → EN-зеркало → линтер
`--against-head` → сайт-гейты.

**Гейты сцен** (Task 2 вносит их в `references/checklist.md` скилла; до этого
источник — список ниже):

1. сцена **заменяет** прозаический пересказ боли; длина ≤ ×1.5 заменённого
   (по словам);
2. сцена сильнее заменяемого открытия — иначе форма ниже по лестнице (§4.3);
3. каждая сцена несёт типизированную функцию (§5.1) — «просто мило» не
   принимается;
4. реплики не пересказывают stdout и числа вывода;
5. реплики `**Ты:**` и ремарки — гендер-нейтральны (линт-предупреждения
   разобраны);
6. число реплик именованных персонажей RU == EN (линт-ошибка);
7. forward-обещания — только прозой; каждый setup — в реестре §7 канона;
8. юмор без каламбуров и эмодзи, переживает перевод;
9. владение концептом не мигрирует (§5.7): сцена подводит, объясняет
   юнит-владелец;
10. персонажи данных (Алиса/Борис/Карина, бариста, Дина/Егор) реплик не
    получают;
11. реплики — только в слотах §4.1 (открытие / перекур / мостик / закрытие
    модуля); внутри теории и `## Запуск` — запрещены (линт-ошибка);
12. ратификации §9 канона подачи — жёсткие входы: развод тёзок прозой (§9.1),
    дедуп называется «по `outbox_id` (таблица `processed_outbox_ids`)» (§9.4),
    `amount` заказа #1 — «итог-заголовок vs позиции», 10.50 и 9.70 рядом без
    объяснения не печатаются (§9.7).

**Порядок пилота:** модуль 00 по номерам, затем 02-04 **до** 02-06 (Руслан
вводится в 02-04 и к своему рапорту «Кассы. Обе. 12:41» в 02-06 уже знаком
читателю), затем 04-01.
