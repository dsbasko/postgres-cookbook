# Переделка подачи курса PostgreSQL Cookbook (гладкое чтение, без потери byte-exact)

## Overview

Сделать так, чтобы материал курса **читался гладко** (на уровне соседнего
`kafka-cookbook`) и ученик **плавно погружался** в тему — не нарушая жёсткий
byte-exact gate курса. Это переделка **подачи** (README-проза), а не контента и не
кода.

- **Проблема:** по двум workflow-разборам средняя вовлечённость при чтении —
  **Kafka 4.67 / 5 vs Postgres 4.05 / 5**. Курс не «плохой», он хорош *неровно*:
  холодные открытия сильные, но вторая половина почти каждого урока сухая.
- **Главная причина:** код «вываливается» готовым и комментируется задним числом,
  вместо того чтобы **собираться по стадиям**. Поверх — три дешёвых markdown-пробела
  (диаграммы, сравнительные таблицы, «Заборчики» одной стеной) и плоские концовки.
- **Ключевые выгоды:** все правки — **только README-проза**, поэтому byte-exact
  gate проходится дёшево; единый источник правды (`docs/course-canon.md`) убирает
  повторы и рассинхрон; сквозной нарратив превращает плоские концовки в
  story-handoff'ы.
- **Интеграция:** не трогает движок сайта (`@dsbasko/cookbook-engine`), демо-код,
  схемы и вставленный stdout; работает строго в `i18n/{ru,en}/README.md`.

Подход выбран в брейншторме: **Вариант A «Зафиксировать → прокатить»**.

## Development Approach

- **«Тест» этого плана = per-unit verification gate.** README-проза не покрывается
  unit-тестами; существующие демо-тесты юнитов **не трогаем**. Готовность каждого
  урока подтверждается прогоном gate (см. Verification Strategy). Везде ниже, где
  скилл планирования требует «write tests», читать как **«прогнать gate юнита»**.
- **Атомарная единица — один урок** (пара `i18n/{ru,en}/README.md`). Полностью
  доводим урок до зелёного gate, прежде чем закрывать модуль.
- **Только README-проза.** НЕ трогать: `query.sql`, `cmd/**/main.go`,
  `internal/db/`, `schema.sql`/`db/init.sql`, и вставленный stdout секции
  `## Запуск`. Любое касание запрещённого файла = регрессия gate, откатить.
- **RU-first, затем синхронный EN.** EN зеркалит RU (та же структура, диаграммы,
  таблицы, акценты) — не «новый текст».
- **Маленькие фокусные изменения**, обновлять этот план по ходу (Progress Tracking).
- **Канон — единственный вход.** Каждый агент-переписчик читает `docs/course-canon.md`
  целиком + строку своего урока из Приложения A. Расхождение urok↔канон —
  блокер (⚠️), фиксируем и решаем, канон правим централизованно.
- **Правило эталонов/хорошо-покрытых юнитов.** `08-01` (эталон), `05-02`, `07-01`,
  `03-01` (по Приложению A покрыты) получают **только аддитивные, stdout-нейтральные**
  правки (недостающая диаграмма, расщепление Заборчика буллетами), **никогда**
  структурную переписку. Применяется единообразно ко всем четырём.

## Verification Strategy

> **⚠️ Что gate НЕ ловит автоматически (выяснено plan-review против кода).** Три
> главные риск-точки prose-переписи **не enforced'ятся** существующим gate:
> (1) байт-точность вставленного stdout `## Запуск` — **нет** авто-сверки README с
> `make run` (это ручной шаг); (2) **EN-паритет** — `web-check-coverage` проверяет
> только *наличие* `i18n/en/README.md` (`existsSync`), не контент; (3) **owner-rules**
> §5 — кросс-урочны, никакой инструмент их не сторожит. Существующий gate
> (`db-reset/gen/run/build/coverage`) останется зелёным, **даже если все три
> нарушены**. Поэтому verify ниже делает их **механическими**, а не «на честном
> слове». TOC/заголовок берутся из `course.yaml` и развязаны с прозой README — это
> **не** риск-точка (см. Technical Details).

Для каждого урока (и перед закрытием каждого модуля):

- **Gate-safety — механически (adversarial verify):** в секции `## Запуск` обычно
  **несколько** fenced-блоков (команды ```sh, блок(и) вывода, а у escape-hatch
  двухсессионных юнитов — ещё ASCII-таймлайн/session-блоки).
  1. **Primary (DB-free):** `git diff` урока показывает **ноль изменённых строк
     внутри любого fenced-блока** секции `## Запуск` (ни команды, ни вывод, ни
     таймлайн не переформатированы), и изменения **только** в
     `i18n/{ru,en}/README.md`; запрещённые файлы
     (`query.sql`/`main.go`/`internal/db/`/`schema.sql`) нетронуты. Это основной
     gate, не требует БД.
  2. **Stronger (когда песочница поднята, `docker compose up -d`):** для **того
     конкретного блока, что является stdout демо** (не-`sh`, не-таймлайн
     output-блок) — `make lecture L=<path>` (== `make run`) и байт-сверка stdout с
     этим блоком. Для sqlc-юнитов нужна живая БД; капстоны (модуль 10) `t.Skip` без
     БД.
  3. **Прозо-блоки НЕ перезапускаем:** ASCII-таймлайн / session-блоки (эталон 05-02:
     `xmin=856/863`, «у тебя числа будут другие») намеренно недетерминированы —
     проверяются только git-diff-нулём из п.1, **не** сравнением с `make run`.
- **EN-паритет — структурно:** RU и EN имеют одинаковое число fenced code-блоков,
  одинаковое число таблиц и диаграмм, **одинаковое число заголовков и совпадающую
  последовательность их уровней** (`grep -c '^#'` + последовательность `#`-глубин),
  а **не** дословное совпадение текста заголовков (они переведены: `## Запуск` ↔
  `## Running it`). Семантику ловит ревью, структуру — эта проверка.
- **Верность SQL-стадий:** каждый показанный «стадийный» SQL — честный
  предшественник/подмножество реального `query.sql`; **финальная стадия совпадает с
  тем, что реально исполняется**. Инструментом не проверяется → обязательная
  проверка ревьюера.
- **Faithfulness к канону:** для каждого концепта, который трогает урок, переписчик
  **цитирует строку-владельца §5** и подтверждает «ссылаюсь, не переобъясняю»; скелет
  §6 применён; обязательные артефакты (диаграмма/таблица/буллет-Заборчик/крючок) на
  месте по Приложению A; спайн-хендофф прозой.
- **Детерминизм §8:** нет печати плавающих значений; выигрыши в счётчиках, не
  таймингах; SQLSTATE как факт; двухсессионность — ASCII-таймлайном.
- **Per-unit gate (из CLAUDE.md):** `make lecture L=<path> T=db-reset` идемпотентен;
  `make gen` без diff; `make run` совпадает с README байт-в-байт; `make build`
  зелёный; оба RU+EN присутствуют.
- **Глобально на модуль:** `make web-check-coverage` зелёный; `make
  web-generate-readme-toc` diff-стабилен (тривиально — TOC из `course.yaml`).
- **Rollback (атомарность на уровне урока):** единица отката — один урок (пара
  RU+EN). Если урок не доводится до зелёного — `git checkout` его пары, фиксируем
  ⚠️, **не блокируем остальные уроки модуля**. Модуль помечается `[x]` только когда
  **каждый** его урок прошёл gate индивидуально.

gate-safety/faithfulness до моего ревью. Каждый урок остаётся отдельной проверяемой
единицей.

## Technical Details

**Что физически меняется в каждом уроке (README-проза обоих языков):**

1. **dump→build:** `query.sql` подаётся стадиями/диффом в тексте («показываем v1 →
   меняем одну строку → финал»); сам `internal/db/` и stdout не меняются.
2. **«Читаем вывод обратно»** (`Во-первых… Во-вторых…`) по фактам фиксированного
   stdout.
3. **Диаграмма** (ASCII/markdown) — где требует Приложение A (43/62).
4. **Сравнительная таблица/cheat-sheet** — где требует Приложение A (48/62).
5. **«Заборчик» буллетами** вместо запятая-стены.
6. **Концовка-крючок** (story-handoff прозой) вместо плоского указателя оглавления.
7. **Newcomer-вставки**, типографские акценты, steel-man, decision-card — по §7.

**Порядок секций юнита — канонический скелет §6:** боль-сцена → концепт стадиями →
stdout → читаем вывод обратно → код как подтверждение → Заборчик буллетами → крючок.

**Gate-дешевизна и где реальная экспозиция:** поскольку демо-код и stdout не
трогаются, `make run` и его байт-точный вывод остаются валидными. **TOC и заголовок
страницы берутся из `course.yaml`** (движок даже срезает H1 README), а `course.yaml`
мы не трогаем — значит TOC/title **автоматически стабильны** под любой prose-правкой,
и `web-generate-readme-toc` всегда diff-стабилен тривиально. Единственная реальная
gate-экспозиция prose-переписи — **вставленный fenced-блок `## Запуск`**: его нельзя
переформатировать/переотступить (нет авто-сверки — ловится только механической
проверкой из Verification Strategy). Туда и направляем внимание ревьюера.
