## §1. Сквозной нарратив (спайн)

### 1.1. Линза: «Первый год бэкендером в Brew»

Курс ведётся от лица **читателя-протагониста**: *ты — новый бэкенд-разработчик
Brew*. Каждый модуль — это **спринт-тикет**, который достраивает прошлую фичу.
Меняешься и ты, и твой арсенал: из новичка, который склеивает SQL строками и
роняет пул, ты вырастаешь в инженера, который защищает инварианты в схеме, читает
план вместо догадок и пишет ретрай-петлю. Финал — ты сам открываешь
`PUBLICATION`, передавая поток изменений Brew в соседний `kafka-cookbook`.

К этой основной линзе подмешаны два слоя:

- **Якорь «заказ #1»** (из арки «жизнь одного заказа») — конкретная строка, к
  которой возвращаемся: **заказ #1 клиентки Алисы Ивановой** (две позиции:
  Капучино + Колд брю, `created_at = 2025-01-15 09:00+00`, статус `paid`). Когда
  нужна осязаемость, говорим про эту строку, а не про абстрактный «заказ».
- **Фон роста** (из арки «рост сети») — Brew растёт от одной кофейни к сети и
  дальше к распределённой системе; это **объясняет, почему боль эскалирует** от
  модуля к модулю. Рост держим **качественным** (точка → сеть → распределёнка),
  **не количественным** — точное число кофеен в каноне нигде не зафиксировано, не
  выдумываем.

### 1.2. Правила применения линзы

1. **Бизнес-боль Brew открывает юнит, а не «сегодня твой тикет №…».** Линза —
   это связующая ткань *между* уроками (хендоффы), а сам урок по-прежнему
   открывается инцидентом бизнеса (см. [§3](#3-повторяющиеся-сцены)). «Спринт» —
   рамка перехода, а не замена холодного открытия.
2. **На созерцательных темах рамку держим лёгкой.** Для MVCC (05-02), чтения
   EXPLAIN (06-01), типов (01-03) спринт-метафора звучит натянуто — это
   *понимание*, а не «ты шипишь фичу». Формулировка там: «ты разбираешься,
   *почему*», а не «ты сдал тикет».
3. **Рост идёт по lab/RICH-таблицам, канон почти статичен.** «Схема растёт к
   капстону» — правда про *твой арсенал и учебную поверхность*, а не про сам
   канон Brew (он неприкосновенен ради byte-compat, см. [§4](#4-сущности-brew)).
   Формулировку держим честной.
4. **Концовка — story-handoff, не оглавление.** Плоское «Дальше — юнит NN
   «…»» заменяется крючком: *«ты зашипил X — и при первом же Y всплывает
   вопрос Z»*. Но ссылаться можно только **прозой** (forward-валидация движка
   запрещает зависимость на ещё не вышедший юнит как на сущность — см.
   [`cookbook-engine` ограничения](../CLAUDE.md)).
5. **Якорь «заказ #1» — там, где он честно есть.** На механических юнитах,
   работающих на `*_lab` (00 пул, 06 EXPLAIN, 02 lab-таблицы), заказа #1
   физически нет — там якорь живёт только в прозе открытия/перехода, не в демо.

### 1.4. Финал: два механизма хендоффа в `kafka-cookbook` (не путать!)

В материале хендофф к Kafka описан **двумя разными механизмами** — это сознательно,
но сейчас читается как один. Канон фиксирует разницу:

- **09-03 — transactional outbox (уровень приложения).** Пишем бизнес-факт и
  событие в **одну** транзакцию в таблицу `outbox`; `relay` доставляет наружу с
  гарантией at-least-once; consumer глушит дубли по `outbox_id` через
  `processed_outbox_ids`. Работает на каноне `orders` / `outbox` /
  `processed_outbox_ids`.
- **10-05 — logical replication / CDC (уровень БД).** `CREATE PUBLICATION
  dbz_publication FOR TABLE drinks, articles, customers` + `REPLICA IDENTITY
  FULL` на этих **трёх редко пишущихся справочниках**; Debezium из `kafka-cookbook`
  читает их без переписывания схемы. 10-05 явно подаёт CDC как **альтернативу**,
  которую он противопоставляет ручному outbox-relay.

> Правило формулировки: 09-03 и 10-05 — **два разных способа** отдать изменения
> наружу, а не два шага одного. Где упоминаем оба — называем их различие.

## §3. Повторяющиеся сцены

«Сцена» = именованный инцидент-хук. **Сквозные** (setup в одном юните → payoff в
другом) держат континуити — их трогать осторожно; **локальные** живут внутри
одного юнита.

### Сквозные нити (setup → payoff)

| Сцена | Маршрут | Суть |
|---|---|---|
| Клиент без заказов (Карина) | 01-02 → 04-* → 08-05 | пропавшая строка: INNER/`count(o.id)`/EXISTS/LEFT-vs-LATERAL |
| Ловушка NULL (трёхзначная логика) | 01-02 (тизер) → 03-06 (расплата) → 04-05 (callback) | `NOT IN + NULL` молча возвращает пусто → довод за `NOT EXISTS` |
| Деньги = BIGINT-центы под защитой | 01-01 → 02-04 (CHECK) → 10-02 | float-дыра → CHECK > 0 → темпоральные цены |
| Переезд во второй город / время | 01-03 → 02-02 | timestamptz-момент → вторая точка получает код-ключ `shops` |
| Транзакционная дисциплина | 03-03 → 05 → 10-01 | `BEGIN→проверка→ROLLBACK` → конкуренция → ретрай 40001 |

| Обещание рекурсивного CTE для деревьев | 04-02/04-06 → 08-04 | self-join staff/категории → рекурсия с защитой CYCLE |
| GIN/containment | 06-05 (механика) → 07-* (применение к меню) | `@>`/`?`/`&&` на jsonb и массивах |
| Опасная миграция на горячей таблице | 02-06 (ACCESS EXCLUSIVE, 8 мин) → 06-06 (CONCURRENTLY) | растущий трафик делает любой ALTER опасным |
| dual-write и его разрыв | 05-01 (forward) → 09-03 (outbox) → 09-04 (NOTIFY-контраст) → 10-05 (CDC заменяет) | согласованность за пределами одной транзакции БД |
| CDC-эстафета в `kafka-cookbook` | 09-03 (точка) → 10-05 (открывает publication) | байт-совместимость канона как through-line: «переименуй колонку — relay сломается» |
| Пул и его цена | 00-06 → 05-06 (advisory утёк) → 09-04 (NOTIFY ломается) → 10-04 (PgBouncer) | у соединения есть физическая цена |

## §6. Канонический скелет юнита

Главная причина «сухости» — код **вываливается** готовым и комментируется задним
числом, вместо того чтобы **собираться**. Эталон, который уже всё делает правильно
внутри ограничений детерминизма, — **08-01** (оконные функции). Канонический
порядок секций юнита:

1. **Боль-сцена** (холодное открытие). Инцидент бизнеса Brew из
   [§3](#3-повторяющиеся-сцены), который заставляет *почувствовать* проблему до
   введения концепта. Не «сегодня твой тикет», а конкретная боль.
2. **Концепт собирается в прозе по стадиям.** Идея растёт слоями; на каждой
   стадии — «почему», прежде чем «как».
3. **Детерминированный stdout** (`## Запуск`) — вставлен дословно, как требует
   gate.
4. **«Читаем вывод обратно»** — `Во-первых… Во-вторых… В-третьих…` по фактам
   *фиксированного* вывода (не перепечатывая его). Детерминизм-безопасно и
   превращает byte-exact в актив: числа не плавают → им можно верить.
5. **Код как подтверждение уже рассказанной истории**, не как ввод. `query.sql`
   показываем **стадиями/диффом** (см. [§7](#7-playbook-подачи)); сам
   сгенерированный `internal/db/` и stdout не меняются.
6. **«Заборчик» буллетами**, не одной запятая-стеной. Один буллет — одна
   production-забота (граница DBA).
7. **Концовка-крючок** (story-handoff в следующий юнит, прозой) — см.
   [правило 1.2.4](#12-правила-применения-линзы).

> Применяя скелет, держим в голове: меняем **только README-прозу** обоих языков
> (`i18n/{ru,en}`). Демо-код, `internal/db/`, и вставленный stdout `## Запуск` —
> **не трогаем**, иначе ломается byte-exact gate. RU-first: сперва `ru`, затем
> синхронный `en`.

## §7. Playbook подачи

Переносимые приёмы (все детерминизм-безопасны — это README-проза):

1. **Сборка кода диффом, не дампом.** Покажи `query.sql` v1 → «меняем ровно одну
   строку» → финал. *Пример 04-01:* показать INNER, потом «дописываем одно слово —
   LEFT», чтобы заявленное «отличается одним словом» было *показано*, а не
   утверждено. *Пример 03-02:* OFFSET-запрос → заменить на `WHERE (base_price, id)
   < (...)` → keyset.
2. **Читать вывод обратно** (`Во-первых… Во-вторых…`), включая честное объяснение
   несовершенных/перекошенных чисел.
3. **Холодное открытие сценой + закрытие петли (callback).** Открыть болью из §3,
   *вернуться* к ней в середине и *разрешить по имени* в концовке. *Пример 04-01:*
   «та самая Карина, что пропала в начале, теперь видна в LEFT JOIN».
4. **Боль-потом-инструмент с телеграфным вердиктом.** Показать неуклюжий обходной
   путь, уронить 2–3-словный вердикт, потом назвать фичу. *Пример 07-01:* «Сорок
   nullable-колонок под опции. Миграция на каждую добавку. ALTER в пятницу. Можно.
   Но больно.» → jsonb.
5. **Оператор/опция = вопрос, на который он отвечает**, не его механика. *Пример
   02-03:* «CASCADE — «дети мусор без родителя?», SET NULL — «переживут сироту?»,
   RESTRICT — «нельзя, пока есть дети?»».
6. **Newcomer-вставки как именованные «на пальцах» абзацы**, и каждый жаргон-токен
   (WAL, seq scan, GIN, Bitmap Heap Scan, write-amplification, TOCTOU) получает
   одну строку интуиции при первом упоминании — **до** ссылки «см. NN».
7. **Steel-man отвергнутого варианта.** Честно: «Можно было взять X — тоже
   нормально», потом обоснование выбора. *Пример 04-01:* re-мотивировать FULL JOIN
   («пишут редко — и это честно; но один раз спасает: сверка двух источников»).
8. **Decision-card компрессия.** Вопрос → ответ в три такта. *Пример 01-01:*
   «Деньги на выходе — BIGINT в центах. Дробная цена за грамм — numeric. float —
   никогда.»
9. **Типографский акцент на одном опорном слове абзаца** (`НЕ` схлопывает, `ВНУТРИ`
   окна) — уже доказано в 08-01.
10. **Множества/Венн в ASCII — подписанными боксами-зонами, не «кругами».**
    Перекрывающиеся круги в моноширинном ASCII читаются плохо. Рабочий приём
    (пилот 04-01): три подписанных бокса (только слева / пересечение / только
    справа) с занявшими их строками + проза «INNER берёт среднюю зону, LEFT —
    среднюю+левую…» + рядом таблица «сохраняет / где NULL / когда брать».
    Обобщается на любую диаграмму отношений множеств.

- **Диаграмма** — на каждом пространственном юните. Минимальный обязательный набор

## §8. Правила детерминизма

Курс печатает **byte-exact** stdout (нет `now()`/random/uuid/xid/pid). Это **не
ограничение, а приём доверия**: «числа не плавают между запусками → им можно
верить» (эхо 05-02: «у тебя числа будут другие — важны их отношения»). Где Kafka
полагается на живой вывод, мы делаем так:

- **Печатаем инварианты и производные свойства, не сырые значения:**
  `ctid_changed=t`, `xmin_changed=t`, `duplicates=0`, версия и монотонность uuid
  (не само значение), число строк. Их и нарративим как «честный результат».
- **Выигрыш производительности — в счётчиках, не в таймингах:** «7 страниц вместо
  7353, ноль выброшенных строк», `Index Searches 9 vs 1`, buffers, rows-removed.
  Wall-clock — только прозой с оговоркой про железо (`TIMING OFF, BUFFERS OFF,
  COSTS OFF`, `max_parallel_workers_per_gather=0`, данные через `generate_series`,
  не `random()`).
- **Конкурентные interleavings — ASCII-таймлайном** (эталон 05-02: Терминал A/B,
  «← снимок A прежний: коммит B не виден»), а `make run` — детерминированный
  single-session путь с инвариантами. Две сессии пинуем `\prompt`.
- **Сквозной through-line — производное свойство, не сырое значение.** Для uuidv7
  печатаем версию-цифру и булеву монотонность; для xid — `xmin_changed=t`. Узнаваемой
  нитью становится *свойство*, не значение.
- **Ошибки — SQLSTATE-код как byte-стабильный факт в stdout; имя+история в прозе.**
  «код детерминирован, текст — нет» (02-03). Это само по себе teaching-beat: почему

   расходится только #1. Проверка кода: значение **10.50 уже напечатано в
   замороженном stdout 04-04** (`DISTINCT ON` по `amount DESC`) и захардкожено в его
   `cmd/demo/main.go`; **9.70 — в stdout 01-01 и 04-06** (сумма позиций). Починка
   seed (10.50→9.70) сломала бы байт-точный stdout 04-04 и противоречила бы его
   `main.go` — планом запрещено. **Решение:** трактуем как **намеренную
   денормализацию** — `amount` есть денормализованный итог-заголовок заказа, который
   легитимно дрейфует от суммы позиций (классический паттерн «header total vs
   line-items»). Делаем **явным учебным хуком**: где урок касается обоих чисел — он
   называет, что они отвечают на **разные** вопросы (записанный итог заказа vs
   пересчёт из позиций), и упоминает источники правды / `GENERATED` / триггеры как
   способ синхронизации. **Правило:** не печатать оба числа рядом без этого
   объяснения; якорь 04-02 («чек из четырёх таблиц») печатает только построчные
   `unit_price` — итог-заголовок туда не подмешиваем. Seed **не трогаем**.
