# 04-01 — JOIN: inner / left / right / full

В Brew данные живут раздельно: клиенты — в `customers`, заказы — в `orders`. Каждый сам по себе бесполезен для бизнес-вопроса «кто из клиентов и сколько заказал». Чтобы ответить, таблицы надо **связать** по общему ключу — это и есть `JOIN`.

Самая частая ловушка новичка: взять обычный (`INNER`) `JOIN` для отчёта «все клиенты и их заказы» — и не заметить, что из отчёта тихо выпали клиенты, которые пока ничего не заказали. В Brew такая клиентка есть — Карина: она зарегистрировалась, но заказов ещё не делала. На каком `JOIN` она исчезнет, а на каком останется (с пустым заказом) — ровно про это юнит.

Важная деталь канона: `customers.id` — это `BIGINT`, а `orders.customer_id` — `TEXT` (так задумано: в реальном CDC-потоке заказы и справочник клиентов едут независимо). Поэтому связываем их с явным приведением `c.id::text = o.customer_id` — это нормально и для нашего разговора про `JOIN` не принципиально.

## INNER JOIN — только совпадения

`INNER JOIN` (или просто `JOIN`) оставляет строки, у которых есть пара **с обеих сторон**. Клиент без заказов и заказ без клиента в результат не попадают. Это правильный выбор, когда нужны именно совпавшие пары — «заказы вместе с данными клиента». Но для отчёта «все клиенты» он коварен: молча выкидывает тех, у кого ещё нет заказов.

## LEFT JOIN — все слева, справа если есть

`LEFT JOIN` сохраняет **все** строки левой таблицы, а из правой подставляет совпадение или `NULL`, если пары нет. `customers LEFT JOIN orders` — это «все клиенты, и их заказы, если есть». Карина останется в результате с `order_id = NULL` и `status = NULL`. Это самый частый `JOIN` в приложениях: «покажи сущность и связанные данные, не теряя сущности без связей».

`LEFT JOIN` + проверка `IS NULL` по правой стороне — стандартный приём «найти строки без пары» (анти-join): `... LEFT JOIN orders o ON ... WHERE o.id IS NULL` вернёт клиентов вообще без заказов.

## RIGHT JOIN — то же зеркально

`RIGHT JOIN` — это `LEFT JOIN` наоборот: он сохраняет все строки **правой** таблицы. `orders RIGHT JOIN customers` даёт ровно то же, что `customers LEFT JOIN orders`: все клиенты, заказы — если есть. Именно поэтому `RIGHT` в коде почти не пишут: любой `RIGHT` превращается в `LEFT` перестановкой таблиц, а `LEFT` читается слева направо естественнее. Знать его нужно, чтобы понимать чужой SQL, но в своём почти всегда выбирают `LEFT`.

## FULL JOIN — несовпадения с обеих сторон

`FULL JOIN` сохраняет несовпавшие строки **с обеих сторон сразу**: и левые без пары, и правые без пары. На каноне Brew его не показать честно — каждый заказ ссылается на существующего клиента, так что «осиротевших» строк справа нет, и `FULL` выродился бы в `LEFT`. Поэтому берём лабораторный пример — сверку двух листов пересчёта остатков: зал посчитал напитки `{1, 2}`, склад — `{2, 4}`. `FULL JOIN` по `drink_id` покажет всё: напиток только в зале (`склад = NULL`), только на складе (`зал = NULL`) и посчитанный в обоих. Это типовая задача «свести два источника и подсветить расхождения».

## Что показывает наш код

Четыре запроса в `query.sql`. Три первых отличаются ровно одним словом (`JOIN` / `LEFT JOIN` / `RIGHT JOIN`) на одной и той же паре таблиц:

```sql
-- name: LeftCustomersOrders :many
SELECT c.name AS customer, o.id AS order_id, o.status
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id::text
ORDER BY c.id, o.id;
```

sqlc видит, что после `LEFT JOIN` колонки `orders` могут быть `NULL`, и типизирует их как `pgtype.Int8` / `pgtype.Text` (а у `INNER`-варианта те же колонки — простые `int64` / `string`: совпадение гарантировано). Четвёртый запрос — `FULL JOIN` листов пересчёта; имя напитка берём из `drinks` по `COALESCE(f.drink_id, s.drink_id)` (ключ есть хотя бы с одной стороны).

Подними песочницу (из корня репозитория) и накати канон:

```sh

```

(`T=run` — значение по умолчанию. Изнутри каталога юнита это `make db-reset`, `make run`.)

Вывод:

```
1) INNER JOIN customers↔orders — только совпавшие пары (строк: 3):
   Алиса Иванова    заказ #1 (paid)
   Алиса Иванова    заказ #3 (shipped)
   Борис Петров     заказ #2 (created)
   → Карины тут нет: у неё нет заказов, совпадать не с чем.

2) LEFT JOIN customers←orders — все клиенты, заказ если есть (строк: 4):
   Алиса Иванова    заказ #1   статус paid
   Алиса Иванова    заказ #3   статус shipped
   Борис Петров     заказ #2   статус created
   Карина Сидорова  заказ —    статус NULL
   → Карина осталась: заказа нет → order_id и status = NULL.

3) RIGHT JOIN orders→customers — тот же результат, что LEFT (строк: 4):
   Алиса Иванова    заказ #1   статус paid
   Алиса Иванова    заказ #3   статус shipped
   Борис Петров     заказ #2   статус created
   Карина Сидорова  заказ —    статус NULL
   → RIGHT = LEFT с переставленными таблицами; в коде почти всегда пишут LEFT.

4) FULL JOIN — сверка листов пересчёта (зал {1,2} vs склад {2,4}):
   напиток         зал    склад
   Эспрессо         10        —
   Капучино          5        3
   Колд брю          —        8
   → строки есть с обеих сторон: только в зале, только на складе, в обоих.
```

INNER дал 3 строки (Карина выпала), LEFT и RIGHT — по 4 (Карина осталась с `NULL`), FULL свёл два листа с расхождениями по краям. Один и тот же набор данных, четыре `JOIN` — четыре разных ответа.

## Заборчик

Что мы упростили. Во-первых, `ON`-условие здесь по неиндексированной паре (`c.id::text = o.customer_id`) и на пяти строках это неважно, но на больших таблицах `JOIN` без подходящего индекса по ключу соединения — это либо hash join с прогоном всей таблицы, либо вложенный цикл, и стоимость растёт быстро (как именно сервер выбирает способ соединения и почему важен индекс под ключ — модуль 06). Во-вторых, приведение `c.id::text` в `ON` — следствие того, что в каноне `customer_id` намеренно `TEXT`; в своей схеме под `JOIN` ключи лучше держать одного типа (а ещё лучше — настоящим внешним ключом), тогда и индекс ложится, и приведение не нужно. И `FULL JOIN` в приложениях встречается редко — он почти всегда сигнал «свожу два независимых источника»; внутри одной нормализованной схемы данные обычно связаны направленно, и хватает `LEFT`.

## Что забрать с собой

- `INNER JOIN` оставляет только пары, совпавшие с обеих сторон, — для отчёта «все сущности» он молча теряет строки без пары.
- `LEFT JOIN` сохраняет все строки левой таблицы; нет пары справа → её колонки `NULL` (sqlc типизирует их как nullable).
- `LEFT JOIN ... WHERE правая.ключ IS NULL` — стандартный анти-join «найди строки без пары».
- `RIGHT JOIN` — зеркало `LEFT`; в коде почти всегда пишут `LEFT`, поменяв порядок таблиц.
- `FULL JOIN` сохраняет несовпадения с обеих сторон — это инструмент сверки двух источников, а не повседневный `JOIN` внутри схемы.
